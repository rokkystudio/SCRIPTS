# PING_TUNNEL_SINGBOX

## Назначение документа

Этот файл описывает **интеграцию PingTunnel с sing-box в Android-клиенте** без публикации исходного кода.

Документ ориентирован на архитектуру, где:

- PingTunnel собран в нативные ABI-бинарники и лежит в модуле приложения;
- есть `TunnelService`, который поднимает VPN/TUN;
- есть backend-слой, где одним из вариантов выступает `sing-box`;
- PingTunnel может включаться или выключаться отдельным конфигом.

---

## 1. Цель интеграции

PingTunnel здесь нужен не как отдельная замена `sing-box`, а как **дополнительный транспортный узел**.

Главная идея:

- основной backend строит обычную клиентскую цепочку;
- **PingTunnel включается последним узлом**;
- backend подключается не к удалённому серверу напрямую, а к **локальному listener-у PingTunnel**;
- дальше PingTunnel доставляет этот трафик на удалённую сторону по ICMP.

Именно поэтому в клиентской архитектуре PingTunnel удобнее держать как **sidecar-процесс**, а не вшивать его внутрь конкретного backend-а.

---

## 2. Рекомендуемая модель: PingTunnel как sidecar

Рекомендуемая схема работы такая:

1. Пользователь включает switch `PingTunnel`.
2. Конфиг подключения сохраняет:
    - флаг включения;
    - `serverHost`;
    - `key`;
    - локальный `listenPort`;
    - режим использования.
3. При старте подключения `TunnelService` поднимает локальный `pingtunnel client`.
4. `pingtunnel client` открывает локальный порт на `127.0.0.1:<localPort>`.
5. Дальше backend получает **переписанную цель подключения**:
    - `host = 127.0.0.1`
    - `port = <localPort>`
6. `sing-box` подключается уже к локальному endpoint-у PingTunnel.
7. PingTunnel доставляет трафик к конечной удалённой цели через свой сервер.

Эта схема удобна тем, что она одинаково подходит и для `sing-box`, и для других backend-ов, если они принимают `host/port` из общего `TunnelStartRequest`.

---

## 3. Почему нельзя ограничиться только switch в конфиге

Один switch сам по себе ничего не маршрутизирует.

Нужно, чтобы при старте VPN-сессии приложение:

- реально запустило sidecar-процесс PingTunnel;
- дождалось локального listener-а;
- отдало backend-у уже новый локальный target;
- при остановке корректно завершило PingTunnel вместе с VPN-сессией.

То есть switch — это только часть **конфигурации**.  
Маршрутизация начинается в месте, где собирается и запускается фактический `TunnelStartRequest`.

---

## 4. Почему нужен `PingTunnelStore`

Для PingTunnel удобно иметь отдельное хранилище настроек, например `PingTunnelStore`.

В нём логично хранить:

- `enabled` — глобальный флаг;
- `mode` — режим работы;
- `serverHost` — адрес PingTunnel Server;
- `key` — числовой ключ клиента;
- `listenPort` — локальный порт listener-а.

Такой store не должен запускать процессы и не должен управлять lifecycle VPN.  
Его задача — только хранить настройки, чтобы стартовый слой приложения мог собрать итоговый сценарий подключения.

---

## 5. Возврат `outboundJson` обязателен

Для интеграции с `sing-box` недостаточно переписать только `host/port`.

Если приложение использует **custom outbound JSON**, то `sing-box` берёт endpoint именно из него.  
В таком сценарии нужно вернуть и передавать дальше `outboundJson`, иначе backend продолжит использовать старые `server/server_port`.

Практическое правило:

- если PingTunnel выключен — `outboundJson` остаётся обычным;
- если PingTunnel включён — в `outboundJson` тоже нужно подменить:
    - `server -> 127.0.0.1`
    - `server_port -> <localPort>`

Иначе получаются два разных источника правды:

- одно значение в `host/port`;
- другое значение внутри `outboundJson`.

Для `sing-box` это приводит к тому, что он продолжает подключаться в старую удалённую точку, а не в локальный listener PingTunnel.

---

## 6. Какие поля нужно переписывать для sing-box

Для outbound-конфигураций `sing-box` критичны именно поля адреса конечного upstream-а.

В практической схеме с PingTunnel нужно считать локальный listener новым upstream endpoint-ом.

Значит, после старта PingTunnel backend должен видеть:

- `server = 127.0.0.1`
- `server_port = <localPort>`

Это одинаково важно как для простых прокси-outbound-ов, так и для custom JSON-конфигов, где endpoint задаётся явно.

---

## 7. Два рабочих режима

### Режим A. PingTunnel как последний узел поверх выбранного backend-а

Это основной рекомендуемый режим.

Схема:

```text
App traffic -> TunnelService/TUN -> sing-box -> 127.0.0.1:<localPort> -> pingtunnel client -> ICMP -> pingtunnel server -> конечный upstream
```

Когда использовать:

- есть обычный серверный профиль;
- есть `sing-box` как основной backend;
- PingTunnel должен быть последним transport hop.

### Режим B. PingTunnel как самостоятельный transport с локальным proxy endpoint

Этот режим возможен, но важно правильно понимать его границы.

PingTunnel может поднять локальный listener и работать как локальный proxy transport.  
Но **для системного трафика Android** этого недостаточно: нужен компонент, который будет захватывать и перенаправлять трафик устройства.

В архитектуре приложения такую роль выполняет `TunnelService` с VPN/TUN.

---

## 8. Можно ли работать без sing-box

Да, но только если разделять два разных сценария.

### Без sing-box, но с приложениями, которые сами умеют ходить через прокси

Это возможно.

Если приложение умеет использовать локальный SOCKS5/HTTP endpoint, можно направить его трафик напрямую в локальный listener PingTunnel без полноценного VPN-режима.

### Без sing-box, но для всего трафика устройства

Для этого всё равно нужен **VPN/TUN-сервис**.  
Один PingTunnel не захватывает системный трафик Android сам по себе.

Именно поэтому для сценария “полный трафик устройства” нужен `TunnelService` или другой VPN-слой, даже если конкретно `sing-box` как backend в этот момент не используется.

### Без удалённого PingTunnel Server

Такой режим не получится.

У PingTunnel должен быть удалённый сервер, иначе локальному клиенту некуда доставлять ICMP-транспорт.

---

## 9. Где именно запускать PingTunnel

Практически правильная точка старта — **`TunnelService`**, а не `SingBoxBackend`.

Причины:

- сервис управляет lifecycle VPN-сессии;
- сервис уже знает, когда начинается и заканчивается туннель;
- сервис может исключить собственный пакет из VPN-фильтра, чтобы sidecar-трафик не зациклился обратно в TUN;
- сервис может единообразно работать и с `sing-box`, и с другими backend-ами.

То есть backend должен получить уже подготовленный request, а запуск sidecar-процесса должен оставаться на уровне сервиса.

---

## 10. Что должен делать стартовый пайплайн

При включённом PingTunnel рекомендуемый пайплайн такой:

1. Прочитать конфиг из `PingTunnelStore`.
2. Проверить, что задан `serverHost`.
3. Проверить корректность `key`.
4. Определить конечный удалённый target.
5. Поднять `pingtunnel client`.
6. Дождаться локального listener-а.
7. Переписать `host/port` на `127.0.0.1:<localPort>`.
8. Переписать `outboundJson`, если используется custom outbound.
9. Запустить выбранный backend уже с новым request.
10. При остановке VPN завершить и backend, и PingTunnel.

---

## 11. Что считать корректной интеграцией

Интеграция сделана правильно, если выполняются все условия:

- switch PingTunnel сохраняется отдельно от остального серверного профиля;
- конфиг PingTunnel хранится в отдельном store;
- sidecar стартует до backend-а;
- backend получает локальный target, а не удалённый;
- `outboundJson` переписывается при использовании custom outbound;
- собственный пакет приложения исключён из VPN-фильтра;
- остановка VPN завершает и PingTunnel тоже.

---

## 12. Краткий вывод

Для текущей архитектуры приложения правильная модель такая:

- **PingTunnel — не отдельный backend, а sidecar transport**;
- **`TunnelService` — точка управления lifecycle**;
- **`PingTunnelStore` — точка хранения конфигурации**;
- **`outboundJson` нужно вернуть и переписывать**, если в `sing-box` используются custom outbound-ы;
- **без `sing-box` можно жить только в ограниченных сценариях**, но для полного системного трафика всё равно нужен VPN/TUN-слой;
- **без удалённого PingTunnel Server режим не работает**.
