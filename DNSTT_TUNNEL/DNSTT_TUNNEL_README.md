# DNSTT_TUNNEL_README

## Что это

DNSTT — это DNS-туннель, который передаёт данные между клиентом и сервером через DNS-запросы.

Транспорт между клиентом и публичным recursive resolver может работать в трёх режимах:

- `DoH` — DNS-over-HTTPS;
- `DoT` — DNS-over-TLS;
- `UDP` — обычный DNS по UDP.

На стороне сервера DNSTT работает как **authoritative DNS-сервер** для выделенной зоны туннеля и принимает запросы по `UDP/53`.

## Как устроена схема

Архитектура DNSTT в этой конфигурации выглядит так:

```text
local app -> dnstt-client -> public recursive resolver -> dnstt-server -> remote app
```

Для режима через публичный relay это означает следующее:

1. `dnstt-client` отправляет DNS-сообщения через выбранный транспорт: `-doh`, `-dot` или `-udp`.
2. Публичный resolver пересылает запросы в authoritative-зону туннеля.
3. `dnstt-server` получает эти запросы на `UDP/53` и передаёт TCP-поток дальше в указанный backend, например `127.0.0.1:22`.

## Что важно понимать сразу

### Отдельный DoH-сервер на VPS не нужен

Для DNSTT не требуется поднимать собственный DoH-сервер на VPS. В этой схеме `DoH` и `DoT` используются **между клиентом и публичным resolver**, а не между клиентом и VPS.

То есть серверная часть DNSTT остаётся обычным authoritative DNS-сервером для своей зоны и принимает трафик по `UDP/53`.

### DNSTT не является VPN сам по себе

DNSTT не поднимает `TUN/TAP`, не выдаёт полноценный VPN-интерфейс и не создаёт SOCKS/HTTP-прокси автоматически. Он связывает:

- локальный TCP-порт на клиенте;
- удалённый TCP-порт на сервере.

Из этого уже можно построить:

- SSH-доступ через туннель;
- локальный SOCKS5 через `ssh -D`;
- HTTP/SOCKS-прокси, если на сервере поднят отдельный прокси.

## Поддерживаемые режимы

### DoH

Режим `DoH` даёт наилучшую маскировку на клиентской стороне, потому что локальный наблюдатель видит HTTPS-трафик к публичному DNS-резолверу.

Пример запуска клиента:

```bash
./dnstt-client -doh https://doh.example/dns-query -pubkey-file server.pub t.example.com 127.0.0.1:8000
```

### DoT

Режим `DoT` работает аналогично, но использует DNS-over-TLS.

Пример запуска клиента:

```bash
./dnstt-client -dot dot.example:853 -pubkey-file server.pub t.example.com 127.0.0.1:8000
```

### UDP

Режим `UDP` поддерживается и полезен для тестов, но не скрывает использование DNS-туннеля.

Пример запуска клиента:

```bash
./dnstt-client -udp 9.9.9.9:53 -pubkey-file server.pub t.example.com 127.0.0.1:8000
```

## Требования к домену

Для работы DNSTT нужен домен, в DNS которого можно сделать **делегацию подзоны через `NS`**.

Рабочая схема зоны:

| Тип    | Имя               | Значение            |
|--------|-------------------|---------------------|
| `A`    | `tns.example.com` | IP VPS              |
| `AAAA` | `tns.example.com` | IPv6 VPS, если есть |
| `NS`   | `t.example.com`   | `tns.example.com`   |

Где:

- `tns.example.com` указывает на сервер;
- `t.example.com` является корнем DNS-зоны, выделенной под туннель.

### Почему `t` должен быть коротким

В DNS-сообщении доступное пространство ограничено. Чем длиннее имя зоны, тем меньше полезной нагрузки остаётся для самих данных туннеля. Поэтому короткая метка вроде `t`, `tt` или `p2p` удобнее длинного имени.

### Почему `tns` нельзя делать под `t`

`tns` нельзя размещать внутри `t`, потому что `t.example.com` — это уже отдельная зона, которую DNSTT использует целиком под туннельный DNS-трафик. Если поместить nameserver внутрь неё, например `tns.t.example.com`, получится, что сервер, который должен обслуживать зону `t.example.com`, сам находится внутри этой же зоны. Тогда для поиска адреса `tns.t.example.com` клиенту сначала нужно обратиться к самой зоне `t.example.com`, а она как раз и обслуживается этим же сервером. Это создаёт замкнутую зависимость и может ломать делегацию или делать её нестабильной.

Поэтому правильно так:

* `tns.example.com` — обычный хост с A/AAAA-записью
* `t.example.com` — отдельная делегированная зона под DNSTT

То есть сначала DNS узнаёт, куда идти за зоной `t.example.com`, через внешний и обычный хост `tns.example.com`, и только потом уже начинает работать сам туннель.


## Какие бесплатные домены подходят, а какие нет

### Не подходят

#### DuckDNS

DuckDNS подходит для обычной A/AAAA-схемы, но не подходит для DNSTT, потому что для DNSTT нужна `NS`-делегация отдельной подзоны, а не только имя, указывающее на IP.

#### dedyn.io в режиме dynDNS

Для доменов `*.dedyn.io` недоступна внешняя делегация `NS`, поэтому такой вариант тоже не подходит под DNSTT.

### Подходит

#### eu.org

`eu.org` подходит для сценария с DNSTT, потому что это сервис делегации доменов на собственные nameserver’ы пользователя. В этой модели можно:

1. временно поднять `BIND9`;
2. пройти проверку домена;
3. после делегации освободить `UDP/53`;
4. запустить `dnstt-server` уже на том же домене.

## Базовый сценарий использования

Самый практичный сценарий для личного доступа:

1. сервер DNSTT принимает запросы на `UDP/53`;
2. backend на сервере — `127.0.0.1:22`;
3. клиент DNSTT поднимает локальный TCP-порт, например `127.0.0.1:8000`;
4. поверх этого создаётся локальный SOCKS5 через:

```bash
ssh -N -D 127.0.0.1:7000 -p 8000 127.0.0.1
```

После этого приложения используют `127.0.0.1:7000` как SOCKS5-прокси.

## Практические ограничения

- `UDP/53` на VPS должен быть доступен извне.
- Сервер DNSTT и `BIND9` не могут одновременно слушать один и тот же `UDP/53`.
- Режим `UDP` обнаруживается проще всего.
- `DoH` и `DoT` скрывают содержимое DNS-сообщений от локального наблюдателя, но сам публичный resolver всё равно видит, что через него идёт туннель.
- Если выбранный публичный resolver плохо переносит крупные ответы, может понадобиться уменьшить `-mtu`.

## Рекомендуемая конфигурация

Для этого проекта оптимальная схема такая:

- VPS на Debian;
- отдельный домен под туннель;
- authoritative DNS через `eu.org`;
- `dnstt-server` на `UDP/53`;
- backend `127.0.0.1:22`;
- основной клиентский режим `DoH`;
- запасной режим `DoT`;
- тестовый или аварийный режим `UDP`.

Такой вариант даёт один сервер DNSTT и несколько клиентских профилей без отдельного DoH-сервера на VPS.
